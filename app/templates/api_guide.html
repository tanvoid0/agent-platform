Agent Platform HTTP API

Use this service from any UI or backend. Prefer versioned paths under /api/v1. Legacy unprefixed routes (/processes, /teams) behave the same.

OpenAPI (Swagger) · LLM config · API tokens · Process demo

Overview

You can use this host without the Flow UI. There are several independent surfaces:

  • OpenAI-compatible LLM — /v1/chat/completions, /v1/models, /v1/embeddings, /v1/health (direct to the embedded proxy; no stored process).
  • Single completion helper — POST /api/v1/chat (one round-trip through the same proxy; no process row).
  • Multi-agent orchestration — /api/v1/processes (goal → planner DAG → optional human gates → task execution). Optional projects and workspace APIs for file sandboxes on the server.
  • Planner roster hints — /api/v1/teams — saved team templates; the only required extra field on POST /api/v1/processes is team_template_id so the planner knows which roles to consider.

Authoritative route list and request bodies: OpenAPI.

Base URL and versioning

Same origin as this page, or your deployed host (for example http://127.0.0.1:18410).

  • Versioned REST: /api/v1/processes, /api/v1/teams, /api/v1/workspaces, /api/v1/projects, /api/v1/projects/{id}/files/…, /api/v1/chat, /api/v1/llm-proxy/…
  • Legacy (same behavior): /processes, /teams, /projects, file sandbox under /projects/{id}/workspace/… (deprecated alias for /files/…)
  • LLM proxy (OpenAI-style): /v1/chat/completions etc. — not under /api/v1
API surface (headless checklist)

Everything below works over HTTP with the same deployment; you do not need the Flow SPA.

  • Liveness — GET /health (no auth).
  • Embedded LLM — /v1/chat/completions, /v1/models, /v1/embeddings, /v1/health. Use when you only need a model call and manage prompts yourself.
  • Planner chat — POST /api/v1/chat — one completion via the configured proxy.
  • Processes — GET/POST /api/v1/processes, GET /api/v1/processes/{id}, GET …/events, GET …/stream, POST …/approve, POST …/tasks/{task_id}/review, POST …/cancel, POST …/sync, POST …/retry, POST …/tasks/{task_id}/retry. See /docs for parameters.
  • Projects — GET/POST /api/v1/projects, GET/PATCH/DELETE /api/v1/projects/{id} — grouping within a workspace; pass workspace_id on create.
  • Workspaces — GET/POST /api/v1/workspaces (master key) — top-level tenant. Resolve your workspace from a token: GET /api/v1/me/workspace.
  • API tokens — /api/v1/workspaces/{workspace_id}/api-tokens/ (master key) — mint workspace-scoped bearer tokens for external apps. Dashboard: /tokens.
  • File sandbox — /api/v1/projects/{project_id}/files/… — list/read/write files under the server’s project sandbox; legacy alias …/workspace/….
  • Team templates — GET/POST /api/v1/teams, GET/PATCH/DELETE /api/v1/teams/{id} — define planner rosters; pass team_template_id when starting a process.
  • LLM proxy (admin) — /api/v1/llm-proxy/* — read/update config.yaml, env, and catalog helpers; same auth as other /api/v1 routes when the master key is set.
Authentication (optional)

If AGENT_PLATFORM_MASTER_KEY is set, protected routes require:

Authorization: Bearer <master key or workspace token>

Master key — full admin access (config UI, workspace CRUD, token minting).

Workspace token (agp_…) — scoped to one workspace; mint via /tokens or POST /api/v1/workspaces/{id}/api-tokens/. Resolve tenant: GET /api/v1/me/workspace.

CORS

Set CORS_ALLOW_ORIGINS to a comma-separated list of browser origins (for example http://localhost:3333). Default when unset is * (credentials disabled with wildcard). The Vite dev UI should use same-origin requests via the dev proxy unless you set VITE_API_ORIGIN.

Single-agent chat (stateless)

POST /api/v1/chat — one completion round-trip through the embedded LLM proxy. Does not create a stored process.

JSON body (fields optional except messages):

{
  "messages": [
    { "role": "user", "content": "Hello" }
  ],
  "model": "your-alias",
  "temperature": 0.2,
  "max_tokens": 256
}

Requires AGENT_PLATFORM_MASTER_KEY on the server (same as for multi-agent runs). The embedded LLM proxy must be reachable at LLM_ORCHESTRATOR_BASE_URL (default same process, port 18410).

OpenAI-compatible surface: /v1/chat/completions, /v1/models, /v1/embeddings, /v1/health on this host. Configure providers and config.yaml via Flow → Settings → LLM proxy or /api/v1/llm-proxy/*. This page’s /docs is Agent Platform’s Swagger.

Multi-agent process

Pick a team template id from GET /api/v1/teams (each item has id and name), or create one with POST /api/v1/teams. Then start a run:

POST /api/v1/processes

{
  "goal": "Research and summarize X",
  "auto_approve": false,
  "team_template_id": 1,
  "client_id": "my-app-namespace"
}

team_template_id is required (integer). Put your document text or instructions in goal, or combine with a project_id and workspace paths your app manages.

You may set client_id in the body or send header X-Agent-Platform-Client: my-app-namespace (header wins when both are set). If AGENT_PLATFORM_REQUIRE_CLIENT_ID is enabled, one of them is required.

Then poll GET /api/v1/processes/{id}, approve the DAG when status is approval_required, review tasks if needed, and subscribe to GET /api/v1/processes/{id}/stream (SSE) or GET /api/v1/processes/{id}/events. Task results appear on each task’s output in the tasks array when completed.

Headless API and external microservices

The same process runs with or without the Flow UI build: orchestration is HTTP-only. Deploy with AGENT_PLATFORM_MASTER_KEY, LLM config.yaml under CONFIG_DIR, and LLM_ORCHESTRATOR_BASE_URL pointing at this service’s /v1 when using the embedded proxy.

Pick an integration style:

  • LLM only — call /v1/chat/completions (or POST /api/v1/chat) from your service; you own prompts, tools, and storage.
  • Full orchestration — POST /api/v1/processes with goal, team_template_id, optional project_id, client_id, and auto_approve / review policy; optionally upload inputs via workspace APIs first, then reference paths in the goal.
  • Hybrid — use the platform for long-running DAG runs and keep your own app DB; correlate with client_id and your external ids.

For orchestration, the usual loop is: create process → poll GET /api/v1/processes/{id} (and approve/review endpoints when gates open) → read task output in the response or tail …/events / …/stream.

Server-to-server calls do not need CORS. Browser clients from other origins require CORS_ALLOW_ORIGINS. A runnable example that uses team templates + processes is scripts/external_microservice_example.py in the repository.

Logical client isolation

Processes may store a client_id. List with GET /api/v1/processes?client_id=my-app-namespace. For any operation on a scoped process, send X-Agent-Platform-Client matching that process’s client_id, or the API returns 404 (same as unknown id).

Legacy rows with null client_id remain visible without the header. For hard isolation, use separate deployments or a gateway.

Approval and task review

When status is approval_required, submit the approved graph with POST /api/v1/processes/{id}/approve and JSON {"dag_json": "..."} (or use auto_approve: true on create for trusted environments only).

When a task requires human review, status becomes task_review_required. Use POST /api/v1/processes/{id}/tasks/{task_id}/review with decision: approve, reject, or request_changes (optional output, feedback, instructions).

Events and SSE

GET /api/v1/processes/{id}/events — append-only event log (optional after_id, limit capped at 2000 per request, event_type). For exports or very large logs, page with after_id set to the last row’s id from the previous response.

GET /api/v1/processes/{id}/stream — Server-Sent Events: the server polls the DB about every 0.8s, emits JSON lines for new EventLog rows since the last emitted id, then repeats. Each data line is JSON, for example:

{
  "task_id": 12,
  "type": "trace",
  "content": "...",
  "timestamp": "2026-04-12T12:00:00"
}

The stream ends when:

  • Process status is completed, failed, or cancelled — if no new logs were pending, the server sends one terminal event {"type":"terminal","content":"<status>"} then closes.
  • Process status is approval_required or task_review_required — after any pending log lines, the server sends {"type":"terminal","content":"approval_required"} or ...task_review_required and closes. The client must use GET /api/v1/processes/{id} (and approve/review endpoints) as the source of truth; reconnect SSE after the gate clears if you still want a live tail.

Correctness for DAG shape and task state remains on GET /api/v1/processes/{id}; SSE is a convenience tail, not an approval channel.

Team templates

GET /api/v1/teams — list templates (use id as team_template_id on POST /api/v1/processes). GET /api/v1/teams/{id} — full roster. POST /api/v1/teams — create a template for your own agent definitions.

Process creation always references a template id; there is no null team on the API.

Health

GET /health — no auth; public liveness.
