The Delegation — AI server rack inside office.glb (Blender)
===========================================================

Import starter (hierarchy + exact object names):
  assets/blender-import/td-ai-server-rack-template.gltf
  → Blender: File → Import → glTF 2.0. See assets/blender-import/README.txt.

Goal: ONE rack in the office file with THREE logical “decks” (LLM, backend, database) as separate
Blender objects. The web app tints each deck via emissive; optional glTF animations are driven
from React (clip names you choose in Blender, with a required prefix).

ROOT (required for embedded mode)
---------------------------------
Create an Empty (or parent object) named exactly:

  TD_AI_Server_Rack

Parent the entire rack hierarchy under it. Position/rotate this object where the rack should sit
in the office. The runtime does NOT move this root when embedded (placement heuristics are only used
for the procedural fallback when this node is missing).

DECK WORKSTATIONS (recommended)
-------------------------------
Under the root, create three empties (or groups) so each tier can hold its own meshes/materials:

  TD_Deck_LLM
  TD_Deck_Backend
  TD_Deck_Database

Put the geometry for each tier under the matching empty. Materials should support emissive (Principled
BSDF Emission). The app clones materials per mesh on load and drives emissive from UI state.

VFX MOUNTS (optional but recommended)
-------------------------------------
Empties parented under TD_AI_Server_Rack (local transform = where you want overlays):

  TD_VFX_SmokeOrigin   — parent for smoke puffs (Ollama error state)
  TD_VFX_StatusLight   — parent for point lights (connection tint)
  TD_VFX_LabelMount    — parent for the floating “AI SERVER” canvas label

If an empty is missing, the runtime falls back to the rack root (functional but less precise).

COLORS / OFFICE TINT
--------------------
Meshes under TD_AI_Server_Rack are excluded from automatic office-wide surface tinting so your
Blender colors stay authoritative. Match the stock rack look with neutral greys (#d4d8df chassis,
#3a4149 rails, #1c2128 vent slots) if you want parity with the procedural fallback.

ANIMATIONS (optional)
---------------------
Actions that should be playable from the app MUST be named with this prefix:

  TD_Rack_

Examples: TD_Rack_FanLoop, TD_Rack_LLM_Pulse, TD_Rack_DB_Blink

Only clips with that prefix are registered on the rack AnimationMixer (other office animations are
ignored). In the UI, add the exact action names to the looping list (Simulation toolbar when Ollama
backend is active), or from code:

  useServerRackPresentationStore.getState().setLoopingAnimationClips(['TD_Rack_FanLoop']);

Blender export checklist
------------------------
- Apply transforms on meshes where needed.
- Prefer NLA strips or push-down actions so each TD_Rack_* action exports.
- glTF export: include animations; use Draco if your pipeline already does for office.glb.
- Append / merge the rack into the main office.blend, then export a single office.glb.
